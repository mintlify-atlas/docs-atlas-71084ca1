Clinical records (`historias_clinicas`) are the structured medical histories MELIKA generates per appointment. Each record is tightly coupled to a single appointment (`id_cita` is UNIQUE), ensuring a one-to-one relationship between a consultation and its clinical narrative. Only authenticated physicians (`isMedico`) may create or add notes; patients may read their own histories and manage the visibility of their own uploaded documents. All endpoints require a valid JWT (`verifyToken`).

---

## Data Model

The `historias_clinicas` table captures the complete clinical encounter — from chief complaint and anamnesis through vital signs, ICD-10 diagnosis, and the prescribing physician's legal signature block. The table supports three record types via `tipo_registro`: a primary history (`historia_principal`) and two note types (`nota_aclaracion`, `nota_evolucion`). Notes link back to their originating record through `id_historia_original`.

| Field | Type | Notes |
|---|---|---|
| `id` | integer | Primary key |
| `id_paciente` | integer | FK → patients |
| `id_medico` | integer | FK → doctors |
| `id_cita` | integer | FK → appointments — UNIQUE per `historia_principal` |
| `tipo_registro` | VARCHAR | `historia_principal`, `nota_aclaracion`, or `nota_evolucion` |
| `estado` | VARCHAR | `activo` or `anulado_por_aclaracion` |
| `id_historia_original` | integer | FK → parent history (set on notes only) |
| `tipo_consulta` | VARCHAR | `presencial` or `teleconsulta`. Default `presencial` |
| `eps_aseguradora` | VARCHAR | Health insurer name |
| `contacto_responsable_nombre` | VARCHAR | Name of responsible contact |
| `contacto_responsable_telefono` | VARCHAR | Phone of responsible contact |
| `motivo_consulta` | TEXT | **Required.** Chief complaint |
| `anamnesis` | TEXT | Medical history narrative |
| `antecedentes_patologicos` | TEXT | Pathological history |
| `antecedentes_quirurgicos` | TEXT | Surgical history |
| `antecedentes_alergicos` | TEXT | Allergy history |
| `antecedentes_familiares` | TEXT | Family medical history |
| `antecedentes_ginecoobstetricos` | TEXT | Gyneco-obstetric history |
| `habitos` | TEXT | Lifestyle habits (smoking, alcohol, etc.) |
| `tension_arterial_sistolica` | FLOAT | Systolic blood pressure (mmHg) |
| `tension_arterial_diastolica` | FLOAT | Diastolic blood pressure (mmHg) |
| `frecuencia_cardiaca` | integer | Heart rate (bpm) |
| `frecuencia_respiratoria` | integer | Respiratory rate (breaths/min) |
| `temperatura_corporal` | FLOAT | Body temperature (°C) |
| `peso_kg` | FLOAT | Weight in kilograms |
| `talla_cm` | FLOAT | Height in centimetres |
| `imc` | FLOAT | BMI — auto-calculated from `peso_kg` / `(talla_cm/100)²` |
| `exploracion_por_sistemas` | TEXT | Systems review findings |
| `examen_fisico` | TEXT | Physical examination findings |
| `diagnostico_cie10` | VARCHAR(10) | ICD-10 diagnosis code (uppercased on save) |
| `descripcion_diagnostico` | TEXT | Human-readable diagnosis description |
| `plan_tratamiento` | TEXT | Treatment plan |
| `medicamentos_recetados` | JSONB | Free-text prescription stored as `{ "texto": "..." }` |
| `ordenes_medicas` | TEXT | Medical orders |
| `recomendaciones` | TEXT | Patient recommendations |
| `incapacidad_dias` | integer | Days of medical leave granted |
| `observaciones` | TEXT | Additional observations |
| `medico_nombre_firma` | VARCHAR | **Required.** Signing physician's full name |
| `medico_cedula_firma` | VARCHAR | Signing physician's ID number |
| `medico_rethus_firma` | VARCHAR | **Required.** Physician's RETHUS registry number |
| `created_at` | timestamp | Auto-generated |
| `updated_at` | timestamp | Auto-updated |

### `medicamentos_recetados` Format

Prescribed medications entered in the free-text clinical notes field are normalised and stored as a JSONB object with a single `texto` key. The API returns the value already unwrapped to the plain string.

```json
{ "texto": "Ibuprofeno 400 mg cada 8 h por 5 días; Omeprazol 20 mg en ayunas" }
```

Individual prescription line items with dosage details are tracked in the separate `recetas_medicas` table via the `recetas` array in the request body (see below).

## Endpoints

### GET /historias/paciente/:id\_paciente

Retrieves a summary list of all primary clinical records (`tipo_registro = historia_principal`) belonging to a specific patient. Each record is enriched with the physician's full name, specialty, appointment date, and a count of attached notes. Patients may only request their own records; physicians may access records for patients with whom they share at least one appointment.

**Authentication:** `verifyToken`

<ParamField path="id_paciente" type="integer" required>
  The unique identifier of the patient whose clinical records are being requested.
</ParamField>

<ResponseField name="[].id" type="integer">
  Clinical record identifier.
</ResponseField>
<ResponseField name="[].motivo_consulta" type="string">
  Chief complaint recorded during the consultation.
</ResponseField>
<ResponseField name="[].diagnostico_cie10" type="string">
  ICD-10 code assigned by the physician.
</ResponseField>
<ResponseField name="[].descripcion_diagnostico" type="string">
  Human-readable diagnosis description.
</ResponseField>
<ResponseField name="[].tipo_registro" type="string">
  Always `historia_principal` for records returned by this endpoint.
</ResponseField>
<ResponseField name="[].estado" type="string">
  Record state: `activo` or `anulado_por_aclaracion`.
</ResponseField>
<ResponseField name="[].total_aclaraciones" type="integer">
  Count of attached clarification/evolution notes.
</ResponseField>
<ResponseField name="[].medico_nombre" type="string">
  First name of the physician who created the record.
</ResponseField>
<ResponseField name="[].medico_apellido" type="string">
  First surname of the physician.
</ResponseField>
<ResponseField name="[].especialidad" type="string">
  Medical specialty of the attending physician.
</ResponseField>
<ResponseField name="[].fecha" type="string">
  Appointment date (`YYYY-MM-DD`).
</ResponseField>
<ResponseField name="[].hora_inicio" type="string">
  Appointment start time.
</ResponseField>

<CodeGroup>
```bash cURL
curl -X GET http://localhost:3000/historias/paciente/42 \
  -H "Authorization: Bearer <token>"
```

```js JavaScript
const res = await fetch('http://localhost:3000/historias/paciente/42', {
  headers: { Authorization: 'Bearer <token>' }
});
const records = await res.json();
```
</CodeGroup>

### GET /historias/cita/:id\_cita

Returns the primary clinical record associated with a specific appointment, together with its attached clarification/evolution notes, prescription lines (`recetas`), and exam orders (`examenes`). If no record exists for the appointment yet, all arrays are returned empty.

**Authentication:** `verifyToken`

<ParamField path="id_cita" type="integer" required>
  The appointment ID for which to retrieve the clinical record.
</ParamField>

<ResponseField name="historia" type="object">
  The primary clinical record object (all `historias_clinicas` columns) enriched with patient and physician demographics, specialty name, and appointment metadata (`fecha_cita`, `hora_inicio`, `tipo_cita`). Returns `null` if no record exists yet.
</ResponseField>
<ResponseField name="aclaraciones" type="array">
  Ordered list of `nota_aclaracion` and `nota_evolucion` records linked to this history.
</ResponseField>
<ResponseField name="recetas" type="array">
  Prescription line items from `recetas_medicas` for this history.
</ResponseField>
<ResponseField name="examenes" type="array">
  Exam orders from `ordenes_examenes` for this history.
</ResponseField>

<CodeGroup>
```bash cURL
curl -X GET http://localhost:3000/historias/cita/88 \
  -H "Authorization: Bearer <token>"
```

```js JavaScript
const res = await fetch('http://localhost:3000/historias/cita/88', {
  headers: { Authorization: 'Bearer <token>' }
});
const { historia, aclaraciones, recetas, examenes } = await res.json();
```
</CodeGroup>

### POST /historias

Creates a new primary clinical record for an appointment. The record is permanently linked via `id_cita`. The controller runs full professional validation (vital-sign ranges, required clinical fields, patient-age coherence) before persisting. If `recetas` or `examenes` arrays are provided they are inserted transactionally in the same request. On success, the linked appointment's `estado` is automatically updated to `completada`.

  The appointment this record belongs to. Must not already have an associated primary record.
</ParamField>
<ParamField body="motivo_consulta" type="string" required>
  Chief complaint — the patient's primary reason for the consultation.
</ParamField>
<ParamField body="anamnesis" type="string" required>
  Medical history narrative gathered during the consultation.
</ParamField>
<ParamField body="antecedentes_alergicos" type="string" required>
  Allergy history.
</ParamField>
<ParamField body="examen_fisico" type="string" required>
  Findings from the physical examination.
</ParamField>
<ParamField body="diagnostico_cie10" type="string" required>
  ICD-10 diagnosis code (max 10 characters). Stored uppercased.
</ParamField>
<ParamField body="descripcion_diagnostico" type="string" required>
  Plain-language description of the diagnosis.
</ParamField>
<ParamField body="plan_tratamiento" type="string" required>
  Recommended treatment plan.
</ParamField>
<ParamField body="medico_nombre_firma" type="string" required>
  Full name of the signing physician as it should appear on the record.
</ParamField>
<ParamField body="medico_rethus_firma" type="string" required>
  Signing physician's RETHUS registry number.
</ParamField>
<ParamField body="antecedentes_patologicos" type="string">
  Pathological history.
</ParamField>
<ParamField body="antecedentes_quirurgicos" type="string">
  Surgical history.

  Health insurer name.
</ParamField>
<ParamField body="contacto_responsable_nombre" type="string">

</ParamField>
<ParamField body="peso_kg" type="number">
  Weight in kilograms. Used to auto-calculate `imc`.
</ParamField>
<ParamField body="talla_cm" type="number">
  Height in centimetres. Used to auto-calculate `imc`.
</ParamField>
<ParamField body="exploracion_por_sistemas" type="string">

Appends a clarification or evolution note (`nota_aclaracion` / `nota_evolucion`) to an existing primary clinical record. Both this route and `PUT /historias/:id` map to the same `actualizarHistoria` controller and accept the same request body. Only the original authoring physician may add notes.

<ParamField body="anamnesis" type="string">

<ParamField body="examen_fisico" type="string">

### PATCH /historias/gestionar-cita/:id

Allows a physician to update the status of one of their appointments to `completada` or `no_asistio`, optionally recording medical notes.

**Authentication:** `verifyToken` + `isMedico`

<ParamField path="id" type="integer" required>
  The appointment ID to manage.
</ParamField>
<ParamField body="estado" type="string" required>
  New appointment status. Accepted values: `completada`, `no_asistio`.
</ParamField>
<ParamField body="notas_medicas" type="string">
  Optional clinical notes to record alongside the status update.
</ParamField>

<ResponseField name="mensaje" type="string">
  `"✅ Cita completada."` or `"📋 Paciente ausente."`
</ResponseField>

<Expandable title="Error Responses">
  | Status | Condition |
  |---|---|
  | `400` | `estado` is not `completada` or `no_asistio`, or the appointment is already `cancelada` |
  | `403` | Appointment does not belong to the requesting physician |
  | `404` | Appointment not found |
</Expandable>

<CodeGroup>
```bash cURL
curl -X PATCH http://localhost:3000/historias/gestionar-cita/88 \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"estado": "completada", "notas_medicas": "Paciente estable al cierre."}'
```

```js JavaScript
const res = await fetch('http://localhost:3000/historias/gestionar-cita/88', {
  method: 'PATCH',
  headers: {
    Authorization: 'Bearer <token>',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({ estado: 'no_asistio' })
});
const { mensaje } = await res.json();
```
</CodeGroup>

## Document Endpoints

Clinical documents (lab results, imaging reports, referrals, external records) are stored separately from structured histories in the `documentos_clinicos` table and linked to patients by `id_paciente`. Patients may only upload `documento_externo` type; physicians may upload `formula_medica`, `orden_examen`, and `historia_clinica` types. Medical documents (`origen = 'medico'`) are immutable — they can never be hidden or deleted.

### GET /historias/documentos/paciente/:id\_paciente

Lists all clinical documents on file for a patient. When the requesting user is a patient, documents where `oculto_paciente = TRUE` are automatically excluded from the response.

### POST /historias/documentos

Registers a new clinical document. Required fields depend on the caller's role: patients may only register `documento_externo` for themselves; physicians may register `formula_medica`, `orden_examen`, or `historia_clinica` for patients under their care.

**Authentication:** `verifyToken`

<ParamField body="id_paciente" type="integer" required>
  The patient this document belongs to.
</ParamField>
<ParamField body="tipo_documento" type="string" required>
  Document type. Patient-allowed: `documento_externo`. Physician-allowed: `formula_medica`, `orden_examen`, `historia_clinica`.
</ParamField>
<ParamField body="url_pdf" type="string" required>
  Full URL pointing to the stored document file (e.g. a cloud storage URL).
</ParamField>
<ParamField body="id_historia" type="integer">
  ID of the linked clinical history (optional).
</ParamField>
<ParamField body="nombre_archivo" type="string">
  Original filename of the document.
</ParamField>
<ParamField body="descripcion" type="string">
  A brief human-readable description of the document contents.
</ParamField>

<ResponseField name="mensaje" type="string">
  `"Documento registrado."`
</ResponseField>
<ResponseField name="documento" type="object">
  The newly created document object including its auto-generated `id` and all persisted fields.
</ResponseField>

<CodeGroup>
```bash cURL

    "tipo_documento": "documento_externo",
```

### PATCH /historias/documentos/:id/ocultar

Allows a patient to hide one of their own externally uploaded documents (`origen = 'paciente'`). Once hidden, the document remains in the database but is excluded from patient-facing document lists. Physician-uploaded documents (`origen = 'medico'`) cannot be hidden.

**Authentication:** `verifyToken` + `isPaciente`

<ParamField path="id" type="integer" required>
  The ID of the document to hide. Must belong to the authenticated patient and have `origen = 'paciente'`.
</ParamField>

<ResponseField name="mensaje" type="string">
  `"Ocultado."`
</ResponseField>

<Info>
  Hiding is a one-way patient action — it sets `oculto_paciente = TRUE`. There is no unhide route. Physicians and administrators may still access hidden documents through privileged views.
</Info>

<CodeGroup>
```bash cURL
curl -X PATCH http://localhost:3000/historias/documentos/7/ocultar \
  -H "Authorization: Bearer <token>"
```

```js JavaScript
const res = await fetch('http://localhost:3000/historias/documentos/7/ocultar', {
  method: 'PATCH',
  headers: { Authorization: 'Bearer <token>' }
});
const { mensaje } = await res.json();
```
</CodeGroup>
