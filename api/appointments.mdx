The Appointments API powers the complete booking lifecycle in MELIKA. Patients can reserve time slots with a specific doctor and specialty, view their upcoming and past appointments, render a FullCalendar-compatible calendar, cancel a booking with a stated reason, and permanently delete cancelled records. Every route is protected by JWT authentication, and the database layer enforces slot availability with concurrency-safe triggers that prevent double-booking even under simultaneous requests.

---

## State Machine

Every appointment begins in `pendiente` and moves through a strict set of transitions. Patients control cancellation and deletion; doctors and administrators control completion and no-show recording.

| Current State | Allowed Transition | Triggered By |
|---|---|---|
| `pendiente` | → `completada` | Doctor / Admin |
| `pendiente` | → `no_asistio` | Doctor / Admin |
| `pendiente` | → `cancelada` | Patient / Admin |
| `cancelada` | → *(deleted)* | Patient |

<Note>
  Once an appointment is marked `completada` or `no_asistio` it cannot be modified or cancelled through the patient-facing API. Only `cancelada` appointments may be permanently deleted.
</Note>

---

## Authentication

All `/citas` routes require a valid JWT passed in the `Authorization` header via the `verifyToken` middleware. The authenticated user's identity is used automatically — patients always operate on their own data only.

```bash
Authorization: Bearer <token>
```

## Endpoints

### <span style={{color:'#16a34a', fontWeight:700}}>POST</span> `/citas` — Create Appointment

Books a new appointment for the authenticated patient. The server looks up the doctor's `tarifa` (fee) automatically and inserts the record with `estado = 'pendiente'`. The target `franja_horaria` (time slot) is locked during the insert to prevent concurrent double-bookings.

<Warning>
  **Concurrency Conflict (PgSQL error 45002):** If two patients attempt to book the same slot simultaneously, the database raises error code `45002`. Your client should catch this `CONCURRENCY_CONFLICT` error and prompt the user to choose a different available slot — do **not** retry the same slot automatically.
</Warning>

**Authentication:** `Authorization: Bearer <token>` — required

#### Request Body

<ParamField body="id_medico" type="integer" required>
  Unique identifier of the doctor to book. The doctor must exist and have an active status; otherwise a `404 NOT_FOUND` is returned.
</ParamField>

<ParamField body="id_especialidad" type="integer" required>
  Identifier of the medical specialty for this appointment.
</ParamField>

<ParamField body="id_franja" type="integer" required>
  Identifier of the time slot (`franja_horaria`) to reserve. The slot must exist and be currently available; if it does not exist or is already taken a `400 INVALID_SLOT` error is returned.
</ParamField>

<ParamField body="fecha" type="string" required>
  Appointment date in `YYYY-MM-DD` format.
</ParamField>

<ParamField body="hora_inicio" type="string" required>
  Appointment start time in `HH:MM` (24-hour) format.
</ParamField>

<ParamField body="tipo_consulta" type="string" default="presencial">
  Consultation modality. Accepted values: `presencial` or `teleconsulta`.
</ParamField>

<ParamField body="motivo" type="string">
  Optional free-text reason or chief complaint for the appointment.
</ParamField>

#### Response — `201 Created`

<ResponseField name="mensaje" type="string">
  Human-readable confirmation message: `"Cita reservada exitosamente."`
</ResponseField>

<ResponseField name="cita" type="object">
  The newly created appointment record.
  <Expandable title="cita fields">
    <ResponseField name="id" type="integer">Unique appointment identifier.</ResponseField>
    <ResponseField name="id_paciente" type="integer">Authenticated patient's identifier.</ResponseField>
    <ResponseField name="id_medico" type="integer">Doctor's identifier.</ResponseField>
    <ResponseField name="id_especialidad" type="integer">Specialty identifier.</ResponseField>
    <ResponseField name="id_franja" type="integer">Reserved time slot identifier.</ResponseField>
    <ResponseField name="fecha" type="string">Appointment date (`YYYY-MM-DD`).</ResponseField>
    <ResponseField name="hora_inicio" type="string">Start time (`HH:MM`).</ResponseField>
    <ResponseField name="tipo_consulta" type="string">`presencial` or `teleconsulta`.</ResponseField>
    <ResponseField name="motivo" type="string">Chief complaint or visit reason, if provided.</ResponseField>
    <ResponseField name="tarifa" type="number">Doctor's consultation fee, resolved automatically from the doctor's profile.</ResponseField>
    <ResponseField name="estado" type="string">Initial state, always `"pendiente"`.</ResponseField>
    <ResponseField name="created_at" type="string">ISO 8601 creation timestamp.</ResponseField>
  </Expandable>
</ResponseField>

#### Error Codes

| HTTP Status | Code | Cause |
|---|---|---|
| `400` | `BAD_REQUEST` | One or more required body fields are missing. |
| `400` | `INVALID_SLOT` | The `id_franja` does not exist or is no longer available (PgSQL `45001`). |
| `404` | `NOT_FOUND` | Doctor with `id_medico` not found or marked inactive. |
| `409` | `CONCURRENCY_CONFLICT` | Another patient booked the same slot simultaneously (PgSQL `45002`). |
| `401` | `UNAUTHORIZED` | JWT missing or invalid. |

#### Example

<CodeGroup>
```bash cURL
curl -X POST http://localhost:3000/citas \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "id_medico": 12,
    "id_especialidad": 3,
    "id_franja": 88,
    "fecha": "2025-09-15",
    "hora_inicio": "09:00",
    "tipo_consulta": "teleconsulta",
    "motivo": "Revisión de resultados de laboratorio"
  }'
```

```json Response 201
{
  "mensaje": "Cita reservada exitosamente.",
  "cita": {
    "id": 201,
    "id_paciente": 45,
    "id_medico": 12,
    "id_especialidad": 3,
    "id_franja": 88,
    "fecha": "2025-09-15",
    "hora_inicio": "09:00",
    "tipo_consulta": "teleconsulta",
    "motivo": "Revisión de resultados de laboratorio",
    "tarifa": 75.00,
    "estado": "pendiente",
    "created_at": "2025-08-01T14:32:00.000Z"
  }
}
```
</CodeGroup>

### <span style={{color:'#2563eb', fontWeight:700}}>GET</span> `/citas/mis-citas` — List My Appointments

Returns all appointments belonging to the authenticated patient, ordered by date and start time (most recent first). Includes doctor name, specialty, and time-slot end time resolved from the `franjas_horarias` table.

**Authentication:** `Authorization: Bearer <token>` — required

#### Request Parameters

No query or path parameters required. The patient is identified from the JWT automatically.

#### Response — `200 OK`

An array of appointment objects. Each element contains:

<ResponseField name="[]" type="array of objects">
  Ordered by `fecha DESC`, `hora_inicio DESC`.
  <Expandable title="Appointment object fields">
    <ResponseField name="id" type="integer">Appointment identifier.</ResponseField>
    <ResponseField name="fecha" type="string">Date in `YYYY-MM-DD` format (formatted via SQL `TO_CHAR`).</ResponseField>
    <ResponseField name="hora_inicio" type="string">Start time from the reserved `franja_horaria`.</ResponseField>
    <ResponseField name="hora_fin" type="string">End time from the reserved `franja_horaria`.</ResponseField>
    <ResponseField name="estado" type="string">Current appointment state: `pendiente`, `completada`, `no_asistio`, or `cancelada`.</ResponseField>
    <ResponseField name="tipo_consulta" type="string">`presencial` or `teleconsulta`.</ResponseField>
    <ResponseField name="motivo" type="string">Visit reason, if recorded.</ResponseField>
    <ResponseField name="tarifa_cobrada" type="number">Fee charged for the appointment.</ResponseField>
    <ResponseField name="razon_cancelacion" type="string">Cancellation reason, populated only when `estado = 'cancelada'`.</ResponseField>
    <ResponseField name="created_at" type="string">ISO 8601 creation timestamp.</ResponseField>
    <ResponseField name="medico_nombre" type="string">Doctor's first name.</ResponseField>
    <ResponseField name="medico_apellido" type="string">Doctor's last name.</ResponseField>
    <ResponseField name="especialidad" type="string">Specialty name.</ResponseField>
  </Expandable>
</ResponseField>

#### Error Codes

### <span style={{color:'#2563eb', fontWeight:700}}>GET</span> `/citas/calendario` — Calendar Events

Returns a list of appointment events formatted for direct consumption by [FullCalendar](https://fullcalendar.io/). `cancelada` appointments are excluded. If no date range is supplied, the response defaults to events within the current calendar month.

**Authentication:** `Authorization: Bearer <token>` — required

#### Query Parameters

<ParamField query="inicio" type="string">
  Start of the date range in `YYYY-MM-DD` format. Defaults to the first day of the current month if omitted.
</ParamField>

<ParamField query="fin" type="string">
  End of the date range in `YYYY-MM-DD` format. Defaults to the last day of the current month if omitted.
</ParamField>

#### Response — `200 OK`

An array of FullCalendar-compatible event objects:

<ResponseField name="[]" type="array of objects">
  <Expandable title="Event object fields">
    <ResponseField name="id" type="integer">Appointment identifier, usable as FullCalendar's event `id`.</ResponseField>
    <ResponseField name="title" type="string">Display label shown on the calendar tile (e.g., doctor name + specialty).</ResponseField>
    <ResponseField name="start" type="string">ISO 8601 datetime string representing the appointment start (`fecha` + `hora_inicio`).</ResponseField>
    <ResponseField name="backgroundColor" type="string">
      Hex color driven by appointment state:
      - `pendiente` → `#B45309` (amber)
      - `completada` → `#1A7A52` (green)
      - `no_asistio` → `#6B7280` (gray)
    </ResponseField>
    <ResponseField name="borderColor" type="string">Same value as `backgroundColor` for visual consistency.</ResponseField>
    <ResponseField name="textColor" type="string">Always `"#fff"` (white) for legibility on all state colors.</ResponseField>
    <ResponseField name="extendedProps" type="object">
      Additional metadata accessible in FullCalendar event handlers.
      <Expandable title="extendedProps fields">
        <ResponseField name="estado" type="string">Appointment state: `pendiente`, `completada`, or `no_asistio`.</ResponseField>
        <ResponseField name="tipo_consulta" type="string">`presencial` or `teleconsulta`.</ResponseField>
        <ResponseField name="medico_nombre" type="string">Full doctor name.</ResponseField>
        <ResponseField name="especialidad" type="string">Specialty name.</ResponseField>
        <ResponseField name="motivo" type="string">Visit reason, if provided.</ResponseField>
        <ResponseField name="tarifa" type="number">Consultation fee.</ResponseField>
      </Expandable>
    </ResponseField>
  </Expandable>
</ResponseField>

<Tip>
  Pass the `inicio` and `fin` parameters directly from FullCalendar's `datesSet` callback to keep the calendar in sync with the user's current view range without over-fetching.
</Tip>

#### Error Codes

| HTTP Status | Code | Cause |
|---|---|---|
| `401` | `UNAUTHORIZED` | JWT missing or invalid. |

#### Example

<CodeGroup>
```bash cURL
curl -X GET "http://localhost:3000/citas/calendario?inicio=2025-09-01&fin=2025-09-30" \
  -H "Authorization: Bearer <token>"
```

```json Response 200
[

    "id": 201,
    "title": "Dr. Carlos Mendoza — Medicina Interna",
    "start": "2025-09-15T09:00:00",
    "backgroundColor": "#B45309",

    "textColor": "#fff",

      "estado": "pendiente",

      "especialidad": "Medicina Interna",
```

### <span style={{color:'#d97706', fontWeight:700}}>PATCH</span> `/citas/:id` — Cancel Appointment

Cancels a `pendiente` appointment owned by the authenticated patient. A `razon_cancelacion` is required. The cancellation atomically updates the appointment state and — via a **database trigger** — automatically sets the corresponding `franja_horaria.disponible = TRUE`, making the slot immediately available for other patients without any additional API call.

<Note>
  **Automatic Slot Liberation:** The database trigger that releases the time slot fires on every `UPDATE` that sets `estado = 'cancelada'`. This means the slot is freed in the same transaction as the cancellation — there is no race window where the slot appears taken after a successful cancellation response.
</Note>

**Authentication:** `Authorization: Bearer <token>` — required

#### Path Parameters

<ParamField path="id" type="integer" required>
  The appointment identifier to cancel.
</ParamField>

#### Request Body

<ParamField body="razon_cancelacion" type="string" required>
  The patient's stated reason for cancelling. Cannot be blank.
</ParamField>

#### Response — `200 OK`

<ResponseField name="mensaje" type="string">
  Confirmation message.
</ResponseField>

<ResponseField name="cita" type="object">
  The updated appointment record reflecting `estado = 'cancelada'` and the recorded `razon_cancelacion`.
  <Expandable title="cita fields">
    <ResponseField name="id" type="integer">Appointment identifier.</ResponseField>
    <ResponseField name="estado" type="string">Updated state: `"cancelada"`.</ResponseField>
    <ResponseField name="razon_cancelacion" type="string">The reason provided in the request.</ResponseField>
    <ResponseField name="fecha" type="string">Appointment date.</ResponseField>
    <ResponseField name="hora_inicio" type="string">Appointment start time.</ResponseField>
    <ResponseField name="id_medico" type="integer">Doctor identifier.</ResponseField>
    <ResponseField name="id_franja" type="integer">Time slot identifier (now freed by the DB trigger).</ResponseField>
  </Expandable>
</ResponseField>

#### Error Codes

| HTTP Status | Code | Cause |
|---|---|---|
| `400` | `BAD_REQUEST` | `razon_cancelacion` missing from request body. |
| `400` | `BAD_REQUEST` | Appointment is already `cancelada`. |
| `400` | `BAD_REQUEST` | Appointment is `completada` and cannot be cancelled. |
| `403` | `FORBIDDEN` | The appointment belongs to a different patient. |
| `404` | `NOT_FOUND` | No appointment found with the given `id`. |
| `401` | `UNAUTHORIZED` | JWT missing or invalid. |

#### Example

<CodeGroup>
```bash cURL
curl -X PATCH http://localhost:3000/citas/201 \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "razon_cancelacion": "Conflicto de horario imprevisto"
  }'
```

```json Response 200
{
  "mensaje": "Cita cancelada correctamente.",
  "cita": {
    "id": 201,
    "id_paciente": 45,
    "id_medico": 12,
    "id_franja": 88,
    "fecha": "2025-09-15",
    "hora_inicio": "09:00",
    "estado": "cancelada",
    "razon_cancelacion": "Conflicto de horario imprevisto"
  }
}
```
</CodeGroup>

### <span style={{color:'#dc2626', fontWeight:700}}>DELETE</span> `/citas/:id` — Delete Appointment

Permanently removes a `cancelada` appointment record belonging to the authenticated patient. Only cancelled appointments may be deleted — this prevents accidental loss of active or completed booking history.

**Authentication:** `Authorization: Bearer <token>` — required

#### Path Parameters

<ParamField path="id" type="integer" required>
  The appointment identifier to delete. Must be in `cancelada` state.
</ParamField>

#### Response — `200 OK`

<ResponseField name="mensaje" type="string">
  Confirmation: `"Cita eliminada correctamente."`
</ResponseField>

#### Error Codes

| HTTP Status | Code | Cause |
|---|---|---|
| `400` | `BAD_REQUEST` | Appointment state is not `cancelada` (e.g., still `pendiente` or `completada`). |
| `403` | `FORBIDDEN` | The appointment belongs to a different patient. |
| `404` | `NOT_FOUND` | No appointment found with the given `id`. |
| `401` | `UNAUTHORIZED` | JWT missing or invalid. |

#### Example

<CodeGroup>
```bash cURL
curl -X DELETE http://localhost:3000/citas/201 \
  -H "Authorization: Bearer <token>"
```

```json Response 200
{
  "mensaje": "Cita eliminada correctamente."
}
```
</CodeGroup>

## Full Booking Flow

The following example walks through the end-to-end lifecycle: creating a new appointment, confirming it appears on the calendar, then cancelling and deleting it.

<Steps>
  <Step title="Book an appointment">
    Reserve a slot by `POST`ing to `/citas`. Store the returned `cita.id` for subsequent calls.

    ```bash
    curl -X POST http://localhost:3000/citas \
      -H "Authorization: Bearer <token>" \
      -H "Content-Type: application/json" \
      -d '{
        "id_medico": 12,
        "id_especialidad": 3,
        "id_franja": 88,
        "fecha": "2025-09-15",
        "hora_inicio": "09:00",
        "tipo_consulta": "teleconsulta",
        "motivo": "Control mensual"
      }'
    # → 201 { "cita": { "id": 201, "estado": "pendiente", ... } }
    ```
  </Step>

  <Step title="Verify on the calendar">
    Fetch the calendar for the target month. The new appointment appears with the `pendiente` amber color (`#B45309`).

    ```bash
    curl -X GET "http://localhost:3000/citas/calendario?inicio=2025-09-01&fin=2025-09-30" \
      -H "Authorization: Bearer <token>"
    # → 200 [{ "id": 201, "backgroundColor": "#B45309", ... }]
    ```
  </Step>

  <Step title="Cancel the appointment">
    Cancel with a reason. The database trigger fires and frees the slot automatically.

    ```bash
    curl -X PATCH http://localhost:3000/citas/201 \
      -H "Authorization: Bearer <token>" \
      -H "Content-Type: application/json" \
      -d '{ "razon_cancelacion": "Ya no es necesario" }'
    # → 200 { "cita": { "estado": "cancelada", ... } }
    ```
  </Step>

  <Step title="Delete the cancelled record">
    Permanently remove the record now that it is in `cancelada` state.

    ```bash
    curl -X DELETE http://localhost:3000/citas/201 \
      -H "Authorization: Bearer <token>"
    # → 200 { "mensaje": "Cita eliminada correctamente." }
    ```
  </Step>
</Steps>
