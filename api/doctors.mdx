The Doctors API covers the full physician lifecycle in MELIKA: administrators register doctors and trigger invitation emails, doctors activate their accounts through a secure token flow, and once active, doctors can access their own profile and record appointment outcomes. All routes share the `/medicos` prefix for admin operations and `/medico` for doctor self-service actions. The server runs on `PORT` (default `3000`).

---

## Account Activation Flow

Before a doctor can log in, an administrator must first create their profile via `POST /medicos`. MELIKA then generates a short-lived invitation token (72-hour expiry) and emails it to the doctor. The doctor visits the activation link and submits their token alongside a chosen password to `POST /medicos/activar`, which sets their password, marks the `usuario` as active and verified, and marks the invitation token as used.

<Info>
  The invitation token is stored in the `tokens_invitacion` table and is single-use. Submitting an expired or already-used token returns an error.
</Info>

```
Admin → POST /medicos → invitation email sent to doctor
Doctor → POST /medicos/activar (token + nueva_password) → account activated
```

## Doctor Activation

### Activate Doctor Account

<Note>
  This is the only public endpoint in the Doctors API — no authentication token is required.
</Note>

```http
POST /medicos/activar
```

Verifies the invitation token against the `tokens_invitacion` table, confirms it has not been used and has not expired, then updates the linked `usuario` record with a bcrypt-hashed password and sets `activo = TRUE`, `verificado = TRUE`. The token is marked as used immediately after a successful activation.

#### Request Body

<ParamField body="token" type="string" required>
  The invitation token delivered to the doctor's email address.
</ParamField>

<ParamField body="nueva_password" type="string" required>
  The password the doctor wants to set for their account. Must be at least 6 characters. Stored as a bcrypt hash.
</ParamField>

#### Response

<ResponseField name="mensaje" type="string">
  Confirmation message: `"Cuenta activada correctamente. Ya puedes iniciar sesión."`
</ResponseField>

#### Errors

| Status | Description |
|--------|-------------|
| `400`  | `token` or `nueva_password` missing, password shorter than 6 characters, token not found, already used, or expired. |
| `500`  | Internal server error. |

#### Example

<CodeGroup>
```bash cURL
curl -X POST http://localhost:3000/medicos/activar \
  -H "Content-Type: application/json" \
  -d '{
    "token": "a3f8c1d2e9b74f05...",
    "nueva_password": "MiContraseñaSegura123!"
  }'
```

```json Response
{
  "mensaje": "Cuenta activada correctamente. Ya puedes iniciar sesión."
}
```
</CodeGroup>

## Admin — Doctor Management

All endpoints in this section require a valid Bearer token from an administrator account. Include the header `Authorization: Bearer <token>` on every request.

### List All Doctors

```http
GET /medicos
```

Returns a flat array of all doctor records, each joined with their linked `usuario` and `especialidad` rows. Useful for building directory listings or admin dashboards.

**Auth:** `verifyToken` + `isAdmin`

### Create Doctor Profile

```http
POST /medicos
```

Creates a new `medico` record and its linked `usuario` shell with a temporary random password, generates a 72-hour invitation token, and dispatches an activation email to the provided address. The doctor completes registration by calling `POST /medicos/activar` with the token they receive.

**Auth:** `verifyToken` + `isAdmin`

#### Request Body

<ParamField body="nombre" type="string" required>
  Doctor's first name.
</ParamField>

<ParamField body="primer_apellido" type="string" required>
  Doctor's first surname.
</ParamField>

<ParamField body="email" type="string" required>
  Email address where the activation invitation will be sent. Must be unique in `usuarios`.
</ParamField>

<ParamField body="tipo_documento" type="string" required>
  Document type. Accepted values: `CC`, `CE`, or `PASAPORTE`.
</ParamField>

<ParamField body="numero_documento" type="string" required>
  Document number. Must be unique in `usuarios`.
</ParamField>

<ParamField body="numero_registro" type="string" required>
  Official medical license or registration number. Must be unique in `medicos`.
</ParamField>

<ParamField body="id_especialidad" type="integer" required>
  ID of the doctor's medical specialty from the `especialidades` table.
</ParamField>

<ParamField body="ciudad" type="string">
  City of practice.
</ParamField>

<ParamField body="acepta_teleconsulta" type="boolean">
  Set to `true` if the doctor offers remote/video consultations. Defaults to `true`.
</ParamField>

<ParamField body="acepta_presencial" type="boolean">
  Set to `true` if the doctor accepts in-person appointments. Defaults to `true`.
</ParamField>

<ParamField body="biografia" type="string">
  Short professional biography displayed on the doctor's public profile.
</ParamField>

<ParamField body="anos_experiencia" type="integer">
  Number of years of professional experience. Defaults to `0`.
</ParamField>

<ParamField body="foto_url" type="string">
  URL of the doctor's profile photo.
</ParamField>

#### Response `201`

<ResponseField name="mensaje" type="string">
  Confirmation that the profile was created and the invitation email was dispatched (e.g. `"Médico creado. Se envió un correo de activación a doctor@clinica.com."`).
</ResponseField>

<Note>
  The `201` response contains only the `mensaje` string. The newly created record is not returned in the body — query `GET /medicos` or `GET /medico/perfil` after activation to retrieve the full profile.
</Note>

#### Errors

| Status | Description |
|--------|-------------|
| `400`  | Missing required fields or `tipo_documento` value is invalid. |
| `401`  | Missing or invalid Bearer token. |
| `403`  | Authenticated user is not an administrator. |
| `409`  | `numero_registro`, `email`, or `numero_documento` already exists. |
| `500`  | Internal server error (email dispatch failures are logged but do not block the `201` response). |

#### Example

<CodeGroup>
```bash cURL
curl -X POST http://localhost:3000/medicos \
  -H "Authorization: Bearer <admin_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "nombre": "Carlos",
    "primer_apellido": "Herrera",
    "email": "carlos.herrera@clinica.com",
    "tipo_documento": "CC",
    "numero_documento": "9876543210",
    "numero_registro": "MED-00456",
    "id_especialidad": 3,
    "acepta_teleconsulta": true,
    "acepta_presencial": false,
    "biografia": "Especialista en medicina interna con 8 años de experiencia.",
    "anos_experiencia": 8
  }'
```

```json Response 201
{
  "mensaje": "Médico creado. Se envió un correo de activación a carlos.herrera@clinica.com."
}
```
</CodeGroup>

### Update Doctor Profile

```http
PUT /medicos/:id
```

Replaces the editable fields on both the `medico` record and its linked `usuario` row. The doctor's email and document details cannot be changed through this endpoint.

**Auth:** `verifyToken` + `isAdmin`

#### Path Parameters

<ParamField path="id" type="integer" required>
  The numeric ID of the `medico` record to update.
</ParamField>

#### Request Body

<ParamField body="nombre" type="string">
  Updated first name.
</ParamField>

<ParamField body="primer_apellido" type="string">
  Updated first surname.
</ParamField>

<ParamField body="ciudad" type="string">
  Updated city.
</ParamField>

<ParamField body="id_especialidad" type="integer">
  Updated specialty ID.
</ParamField>

<ParamField body="tarifa" type="number">
  Updated consultation fee. Defaults to `0` if not provided.
</ParamField>

<ParamField body="acepta_teleconsulta" type="boolean">
  Updated teleconsultation preference.
</ParamField>

<ParamField body="acepta_presencial" type="boolean">
  Updated in-person preference.
</ParamField>

<ParamField body="biografia" type="string">
  Updated biography text.
</ParamField>

<ParamField body="anos_experiencia" type="integer">
  Updated years of experience.
</ParamField>

<ParamField body="foto_url" type="string">
  Updated profile photo URL.
</ParamField>

#### Response

<ResponseField name="mensaje" type="string">
  Confirmation: `"Médico actualizado correctamente."`
</ResponseField>

#### Errors

| Status | Description |
|--------|-------------|
| `401`  | Missing or invalid Bearer token. |
| `403`  | Authenticated user is not an administrator. |
| `404`  | No doctor found with the given ID. |
| `500`  | Internal server error. |

#### Example

<CodeGroup>
```bash cURL
curl -X PUT http://localhost:3000/medicos/7 \
  -H "Authorization: Bearer <admin_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "nombre": "Carlos Alberto",
    "tarifa": 75.00,
    "acepta_presencial": true
  }'
```

```json Response
{
  "mensaje": "Médico actualizado correctamente."
}
```
</CodeGroup>

### Toggle Doctor Active Status

```http
PATCH /medicos/:id/estado
```

Toggles the `activo` boolean on both the `medico` and linked `usuario` records. Deactivating a doctor prevents them from logging in and excludes them from patient-facing search results. Re-activating reverses this.

**Auth:** `verifyToken` + `isAdmin`

#### Path Parameters

<ParamField path="id" type="integer" required>
  The numeric ID of the `medico` record whose status should be toggled.
</ParamField>

#### Response

<ResponseField name="mensaje" type="string">
  `"Médico activado."` if the account was just enabled, or `"Médico desactivado."` if it was just disabled.
</ResponseField>

#### Errors

| Status | Description |
|--------|-------------|
| `401`  | Missing or invalid Bearer token. |
| `403`  | Authenticated user is not an administrator. |
| `404`  | No doctor found with the given ID. |
| `500`  | Internal server error. |

#### Example

<CodeGroup>
```bash cURL
curl -X PATCH http://localhost:3000/medicos/7/estado \
  -H "Authorization: Bearer <admin_token>"
```

```json Response — deactivated
{
  "mensaje": "Médico desactivado."
}
```

```json Response — activated
{
  "mensaje": "Médico activado."
}
```
</CodeGroup>

## Doctor Self-Service

The following endpoints are available to authenticated doctors only. Use the Bearer token issued at login with `rol='medico'`.

### Get Own Profile

```http
GET /medico/perfil
```

Returns the authenticated doctor's full profile by joining their `medico`, `usuario`, and `especialidad` rows. Doctors use this to display their own information within the platform dashboard.

**Auth:** `verifyToken` + `isMedico`

#### Response

A single flat profile object returned directly (not wrapped in a key).

<ResponseField name="id" type="integer">Unique identifier of the `medico` record.</ResponseField>
<ResponseField name="numero_registro" type="string">Official medical registration number.</ResponseField>
<ResponseField name="tarifa" type="number">Consultation fee.</ResponseField>
<ResponseField name="calificacion" type="number">Average rating score.</ResponseField>
<ResponseField name="acepta_teleconsulta" type="boolean">Whether the doctor accepts video/remote consultations.</ResponseField>
<ResponseField name="acepta_presencial" type="boolean">Whether the doctor accepts in-person consultations.</ResponseField>
<ResponseField name="biografia" type="string">Short professional biography.</ResponseField>
<ResponseField name="anos_experiencia" type="integer">Years of professional experience.</ResponseField>
<ResponseField name="foto_url" type="string">Profile photo URL, if set.</ResponseField>
<ResponseField name="nombre" type="string">Doctor's first name (from `usuarios`).</ResponseField>
<ResponseField name="primer_apellido" type="string">Doctor's first surname (from `usuarios`).</ResponseField>
<ResponseField name="email" type="string">Email address (from `usuarios`).</ResponseField>
<ResponseField name="telefono" type="string">Phone number (from `usuarios`).</ResponseField>
<ResponseField name="ciudad" type="string">City (from `usuarios`).</ResponseField>

### Manage Appointment Outcome

```http
PATCH /medico/citas/:id/gestionar
```

Allows the authenticated doctor to record the final outcome of an appointment. Only the two terminal states are accepted: `completada` (the patient attended) and `no_asistio` (the patient did not show up). Cancelled appointments cannot be gestioned.

**Auth:** `verifyToken` + `isMedico`

#### Path Parameters

<ParamField path="id" type="integer" required>
  The numeric ID of the `cita` (appointment) to update.
</ParamField>

#### Request Body

<ParamField body="estado" type="string" required>
  The outcome to record. Accepted values: `"completada"` or `"no_asistio"`. Any other value returns a `400` error.
</ParamField>

<ParamField body="notas_medicas" type="string">
  Optional brief clinical notes for the appointment. If omitted, any existing notes are preserved.
</ParamField>

#### Response

<ResponseField name="mensaje" type="string">
  Confirmation message:
  - `"✅ Cita completada."` when `estado = 'completada'`
  - `"📋 Paciente ausente."` when `estado = 'no_asistio'`
</ResponseField>

#### Errors

| Status | Description |
|--------|-------------|
| `400`  | `estado` value is not `completada` or `no_asistio`. |
| `400`  | Appointment is already `cancelada`. |
| `401`  | Missing or invalid Bearer token. |
| `403`  | Authenticated user has no doctor profile, or the appointment belongs to another doctor. |
| `404`  | No appointment found with the given ID. |
| `500`  | Internal server error. |

#### Example

<CodeGroup>
```bash cURL
curl -X PATCH http://localhost:3000/medico/citas/42/gestionar \
  -H "Authorization: Bearer <medico_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "estado": "completada",
    "notas_medicas": "Paciente estable, se recomienda seguimiento en 30 días."
  }'
```

```json Response
{
  "mensaje": "✅ Cita completada."
}
```
</CodeGroup>
