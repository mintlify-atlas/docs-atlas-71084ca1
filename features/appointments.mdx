MELIKA's appointment system covers the full journey from initial booking through completion or cancellation. Every appointment is anchored to a 40-minute time slot (`franja_horaria`), a specific doctor, and a specialty — ensuring consistent scheduling across both in-person and teleconsultation modalities. The platform enforces concurrency-safe booking, automatic slot liberation on cancellation, and real-time calendar rendering for both patients and medical staff.

## Appointment Lifecycle

Each appointment moves through a set of well-defined states. The flow begins at `pendiente` when a patient books a slot, then resolves to one of three terminal states once the appointment date has passed or an action is taken.

| State | Color | Hex | Meaning |
|---|---|---|---|
| `pendiente` | Amber | `#B45309` | Appointment has been booked and is awaiting the scheduled date |
| `completada` | Green | `#1A7A52` | The appointment was attended and the doctor has managed it |
| `cancelada` | — | — | The appointment was explicitly cancelled before taking place |
| `no_asistio` | Gray | `#6B7280` | The patient did not show up for the scheduled appointment |

<Note>
  State colors map directly to the `backgroundColor` and `borderColor` fields returned by the `GET /citas/calendario` endpoint, so the FullCalendar UI reflects states without any additional client-side mapping.
</Note>

## Consultation Types

When booking, patients choose between two consultation modalities via the `tipo_consulta` field:

<CardGroup cols={2}>
  <Card title="Presencial" icon="hospital">
    In-person visit at the clinic or doctor's office. Selected by passing `tipo_consulta: "presencial"` in the booking request.
  </Card>
  <Card title="Teleconsulta" icon="video">
    Remote video consultation. Selected by passing `tipo_consulta: "teleconsulta"` in the booking request. Doctor availability for this modality is exposed via `acepta_teleconsulta` on the specialty-doctor listing.
  </Card>
</CardGroup>

## How Booking Works

<Steps>
  <Step title="Browse Available Specialties">
    Fetch the active specialties catalog. Only specialties with `activa = TRUE` are returned.

    ```http
    GET /especialidades
    ```

    Returns an array of specialty objects including `id`, `nombre`, `descripcion`, `precio_base`, and `imagen_url`.
  </Step>
  <Step title="Pick a Doctor">
    List all doctors for a chosen specialty. The response includes profile details such as `foto_url`, `tarifa`, `calificacion`, `acepta_teleconsulta`, `acepta_presencial`, `biografia`, and `anos_experiencia`.

    ```http
    GET /especialidades/:id/medicos
    ```
  </Step>
  <Step title="Check Slot Availability">
    Query available `franjas_horarias` for a specific doctor on a given date. Each franja is 40 minutes long.

    ```http
    GET /especialidades/disponibilidad?medico_id=3&fecha=2025-08-15
    ```

    To query across a date range and receive FullCalendar-ready event objects, use the range endpoint instead:

    ```http
    GET /especialidades/disponibilidad-rango?medico_id=3&inicio=2025-08-01&fin=2025-08-31
    ```
  </Step>
  <Step title="Book the Appointment">
    Submit the booking with the required fields. `tipo_consulta` and `motivo` are optional.

    ```http
    POST /citas
    Content-Type: application/json

    {
      "id_medico": 3,
      "id_especialidad": 2,
      "id_franja": 14,
      "fecha": "2025-08-15",
      "hora_inicio": "09:00",
      "tipo_consulta": "presencial",
      "motivo": "Revisión anual de tensión arterial"
    }
    ```

    On success, the full `cita` object is returned including the assigned slot details and initial state `pendiente`.
  </Step>
</Steps>

## Booking Request Parameters

<CardGroup cols={2}>
  <Card title="Required Fields" icon="circle-exclamation">
    - `id_medico` — ID of the selected doctor
    - `id_especialidad` — ID of the specialty
    - `id_franja` — ID of the time slot from `franjas_horarias`
    - `fecha` — Date of appointment (`YYYY-MM-DD`)
    - `hora_inicio` — Start time of the slot (`HH:MM`)
  </Card>
  <Card title="Optional Fields" icon="circle-info">
    - `tipo_consulta` — `"presencial"` or `"teleconsulta"` (defaults to `"presencial"`)
    - `motivo` — Free-text reason for the visit
  </Card>
</CardGroup>

## Viewing Appointments

<Tabs>
  <Tab title="Appointment List">
    Returns all appointments for the authenticated patient, enriched with the doctor's full name, specialty name, and the slot's `hora_fin` from the `franjas_horarias` table.

    ```http
    GET /citas/mis-citas
    ```

    Example response item:

    ```json
    {
      "id": 42,
      "fecha": "2025-08-15",
      "hora_inicio": "09:00",
      "hora_fin": "09:40",
      "estado": "pendiente",
      "tipo_consulta": "presencial",
      "motivo": "Revisión anual de tensión arterial",
      "medico": "Dr. Andrés Ramírez",
      "especialidad": "Cardiología"
    }
    ```
  </Tab>
  <Tab title="Calendar View">
    Returns appointment data shaped as FullCalendar event objects. Each event includes `backgroundColor` and `borderColor` already set to the state's corresponding color code, so no client-side transformation is needed.

    ```http
    GET /citas/calendario
    ```

    Example response item:

    ```json
    {
      "id": 42,
      "title": "Cardiología — Dr. Andrés Ramírez",
      "start": "2025-08-15T09:00:00",
      "end": "2025-08-15T09:40:00",
      "backgroundColor": "#B45309",
      "borderColor": "#B45309",
      "extendedProps": {
        "estado": "pendiente",
        "tipo_consulta": "presencial"
      }
    }
    ```

    Pass this array directly to FullCalendar's `events` prop.
  </Tab>
</Tabs>

## Cancellation Rules

To cancel a `pendiente` appointment, send a `PATCH` request with a mandatory cancellation reason. Cancelling a `completada` appointment is not permitted.

```http
PATCH /citas/:id
Content-Type: application/json

{
  "razon_cancelacion": "El paciente no puede asistir por motivos de viaje"
}
```

<Warning>
  The `razon_cancelacion` field is **required** in the request body. Omitting it will result in a validation error and the appointment will not be cancelled.
</Warning>

When an appointment is successfully cancelled, the associated `franja_horaria` is **automatically freed** via a database trigger, making it immediately available for other patients to book.

## Deleting an Appointment

Only appointments with `estado = 'cancelada'` may be permanently deleted. Attempting to delete an appointment in any other state will return an error.

```http
DELETE /citas/:id
```

<Tip>
  Deletion is a hard delete. If you need to retain a cancellation record for auditing purposes, keep the appointment in `cancelada` state rather than deleting it.
</Tip>

## Concurrency Protection

MELIKA uses database-level concurrency control to prevent double-booking. If two patients attempt to book the same slot at the same time, only one request will succeed. The other will receive a structured error response.

| Error Code | Name | Description |
|---|---|---|
| `45002` | `CONCURRENCY_CONFLICT` | Another user booked the same slot between your availability check and your booking request. Prompt the user to select a different slot. |
| `45001` | `INVALID_SLOT` | The selected `franja_horaria` does not exist or is not available for the requested doctor and date combination. |

```json
{
  "code": "CONCURRENCY_CONFLICT",
  "message": "El horario seleccionado ya fue reservado por otro usuario. Por favor, elige otro turno disponible."
}
```

<Note>
  When a `CONCURRENCY_CONFLICT` error occurs, refresh the availability query (`GET /especialidades/disponibilidad`) to retrieve the updated list of free slots before allowing the user to rebook.
</Note>
